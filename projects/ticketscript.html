Ticket Automation Script

Custom python script built to create and distribute tickets

Overview

I developed an end‑to‑end digital ticket workflow that generates unique, personalised tickets for each attendee and supports simple distribution. The goal was to remove repetitive admin tasks (manual formatting, printing, packaging, posting) while maintaining a professional, branded ticket output. The solution was designed to be low‑cost, repeatable for future events, and easy for non‑technical staff to use once set up.

Project Context

My workplace regularly hosts events, but the ticketing process relied on generic printed tickets and manual admin work to create, package, and post them. This approach took days of effort and limited personalisation. I proposed switching to digital tickets to reduce admin time and improve attendee experience. After researching existing ticketing platforms, I found that full personalisation at scale was costly, so I built a custom solution in Python.

System Architecture

Input: Read a CSV of attendee data (name, email, ticket qty, barcode string).

Generate: For each attendee, create barcode + text overlay, then merge onto a PDF ticket template.

Send: Email tickets via SMTP in batches, refreshing the SMTP session every 10 emails to prevent timeouts.

Implementation

During implementation, the main challenge was SMTP reliability. My first approach opened the SMTP connection once at the start, sent all emails, then closed it at the end. This repeatedly failed because the server timed out mid‑batch, leaving the remaining tickets unsent. I then switched to opening/closing the connection per attendee, which improved reliability but significantly increased runtime.

To balance speed and stability, I implemented a session refresh strategy: the SMTP connection is reused but re-authenticated every 10 emails, preventing timeouts without the overhead of reconnecting for every recipient.

A second issue was recoverability. If the script failed partway through, there was no clear record of progress. To solve this, the script writes back to a tracking CSV every 5 attendees, marking tickets as sent so the process can resume without duplication.

Finally, I improved robustness by preventing single-recipient failures from stopping the entire run. I implemented a try/except workflow so failed sends are logged and the script continues processing the remaining attendees, enabling reliable batch completion and easier fault investigation.

Deployment & Impact

This solution replaced a manual, print‑and‑post ticketing workflow with a fully automated digital system. Previously, ticket distribution involved ordering printed tickets, waiting for production and delivery, packaging envelopes, labelling, posting, and waiting for arrival, resulting in an end‑to‑end turnaround of around two weeks. The automated workflow reduced this entire process to approximately three minutes of execution time, completely eliminating reliance on external printing and postal services.

The change delivered a dramatic efficiency improvement, reducing turnaround time by roughly 99.99% and increasing delivery speed by over 6,700×. In addition to saving time, the system significantly improved the professional quality of tickets through consistent branding and full personalisation. On the day of events, operations were smoother, with automated check‑in enabled through QR or barcode scanning rather than manual name checking.

The digital approach also removed scalability limits inherent in manual distribution, allowing hundreds or thousands of tickets to be generated and sent with no additional administrative overhead. The codebase is fully reusable, enabling the same workflow to be deployed for future events with minimal setup.

Improvements

A key improvement would be to add a simple user interface so the tool no longer relies on hard‑coded values or a developer editing the script. A lightweight UI (desktop or web) could guide staff through selecting the event template, importing the attendee CSV, previewing ticket output, and triggering distribution with clear progress and error reporting. This would make the workflow usable by non‑technical users with minimal training, reduce the risk of input mistakes, and remove the dependency on someone with Python/HTML knowledge. It would also support better governance (standardised templates, controlled settings, and consistent usage across the organisation).
